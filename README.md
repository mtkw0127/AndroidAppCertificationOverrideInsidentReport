# App Certification Override Incident

A sample project demonstrating a security incident caused by the combination of **custom permission
name collision** and **signature rotation (APK Signature Scheme v3)** on Android.

## Overview

A `signature`-level custom permission is designed to be granted only to apps signed with the same
certificate. However, that assumption breaks when all of the following conditions are met:

- Two apps declare a **custom permission with the same name**
- The app installed later uses **signature rotation** with the earlier app's certificate as an
  ancestor

This repository reproduces that scenario in a minimal setup so the behavior can be observed.

## Signing Configuration

```
keystore_ancestor.jks  (ancestorkey)
        │
        │  apksigner rotate
        ▼
keystore2.jks          (app2key)   ← lineage_app2.bin records the ancestor→new key lineage
```

| App  | Package name              | Signing key                  | Note                  |
|------|---------------------------|------------------------------|-----------------------|
| App1 | `io.github.mtkw0127.app1` | `ancestorkey` (ancestor key) | V3 signature only     |
| App2 | `io.github.mtkw0127.app2` | `app2key` (post-rotation)    | V3 + embedded lineage |

## How the Incident Works

1. **Install App1 first** → becomes the owner of the `CUSTOM_PERMISSION` (signature) declaration
2. **Install App2** → also declares `CUSTOM_PERMISSION` with the same name; its APK embeds the "
   ancestor key → app2key" lineage
3. Android consults the lineage and recognizes App2 as a legitimate successor to App1's certificate
4. As a result, App2 is granted `CUSTOM_PERMISSION` **even though it is signed with a different key
   **

## Installation

Apps **must be installed in App1 → App2 order** to reproduce the incident.

```bash
# Install and launch App1
./gradlew :app1:installDebug

# Install and launch App2 (re-signing with lineage is run automatically)
./gradlew :app2:installDebug
```

During App2's build, the `signDebugWithLineage` task runs automatically before `installDebug` and
uses `apksigner` to embed the lineage into the APK.

## Verification

Launch both apps. The screen shows the result of communication via **BroadcastReceiver** and *
*ContentProvider**.

- If `CUSTOM_PERMISSION` is granted: messages arrive successfully
- If not granted: `SecurityException` is displayed

## Reproduction Steps

### Step 1 — Install App1 and verify its signature

```bash
./gradlew :app1:installDebug
```

```bash
adb shell dumpsys package io.github.mtkw0127.app1 | grep signatures=PackageSignatures
```

Expected output — App1 is signed with the ancestor key only:

```
signatures=PackageSignatures{2e84865 version:3, signatures:[c4902d79], past signatures:[]}
```

### Step 2 — Install App2 and re-check both signatures

```bash
./gradlew :app2:installDebug
```

```bash
adb shell dumpsys package io.github.mtkw0127.app1 | grep signatures=PackageSignatures && \
adb shell dumpsys package io.github.mtkw0127.app2 | grep signatures=PackageSignatures
```

Expected output — **App1's recorded signature is now updated to match App2's rotated key**:

```
signatures=PackageSignatures{2e84865 version:3, signatures:[612cf88c], past signatures:[c4902d79 flags: 17, 612cf88c flags: 17]}
signatures=PackageSignatures{d9dcc31 version:3, signatures:[612cf88c], past signatures:[c4902d79 flags: 17, 612cf88c flags: 17]}
```

Because App2's lineage includes the ancestor key, Android treats both apps as having the same
effective signing identity and overwrites the signature record stored for App1.

### Step 3 — Attempt to update App1 (fails)

```bash
./gradlew :app1:installDebug
```

Expected error:

```
INSTALL_FAILED_UPDATE_INCOMPATIBLE: Existing package io.github.mtkw0127.app1 signatures do not match newer version; ignoring!
```

The system now expects App1 to be signed with the rotated key (`612cf88c`) because its record was
overwritten in Step 2. The original ancestor-key-signed APK no longer matches, so the update is
rejected.
